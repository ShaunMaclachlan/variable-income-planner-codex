# VIP next-phase execution proposal

## Decision to challenge

Track A is a deliberately minimal, throwaway-tolerant pilot. It exists to make gross-pay decisions trustworthy and generate evidence. Track B is a clean product build and does not begin in earnest until Track A completes a full pay period and a second committed user brings a different employer, rate card, payslip and source provider.

If no second user materialises, stopping is a valid result. Track A code is evidence, not the product architecture.

## Track A: pilot

### Stage 1 — trust baseline

- Remove real personal data and service credentials from source, build output and automatic deployments.
- Store and calculate money in integer pence with a visible, provisional rounding policy.
- Preserve source times; flag rather than normalise template deviations.
- Never convert planned work to worked because its date passed.
- Separate earned/confirmed, planned and total forecast values.
- Make disappearance and identity ambiguity visible during calendar review.
- Keep calendar links local and out of encrypted backup files.
- Remove pilot cloud sync; provide passphrase-encrypted export/import recovery.
- Preserve historical rates with effective dates.
- Require a date before paid holiday affects assessment or payroll forecasts.

### Stage 2 — smallest usable workflow

- Manual add/edit remains the guaranteed path from a phone.
- Read a source `.ics` feed or file on demand and require review before every application.
- Confirm every status change explicitly. No source change silently affects the forecast.
- Add a pilot-only interpretation form for copied source wording only if it materially reduces entry errors. It is not an email integration and copied content is not retained after confirmation.
- Do not use automatic forwarding. Do not connect a live mailbox in Track A. Restricted Gmail OAuth test tokens expire after seven days and production verification/security costs are disproportionate to a one-user evidence pilot; employer policy may block it entirely.
- If removing copy/paste is a non-negotiable acceptance criterion, pause and make the provider-specific OAuth/security investment an explicit go/no-go decision rather than disguising it as a small feature.
- Calendar publication is separate from calendar reading. If the pilot needs VIP-confirmed shifts to appear in Apple Calendar, use a dedicated read-only subscribed calendar generated by VIP; do not attempt CalDAV writes into an existing personal calendar.

### Stage 3 — evidence run

- Run one complete verified cutoff-to-cutoff pay period; extend if it produces fewer than about ten genuine source changes.
- Compare VIP with an independent manual source log. Zero unexplained differences is the target.
- Reconcile one payslip line by line. The proposed gross tolerance is £1, but rounding evidence may justify tightening it.
- Record every occasion when pull-on-open was insufficient and a push notification was wanted.
- Ask whether gross pay actually answers the decision, or is only a proxy for assessment-period or net impact.

### Safety pause and stop rules

Pause sync and use manual mode for a silent time/status/pay mutation, any new public personal-data exposure, or an unexplained forecast/payslip difference above tolerance.

Stop product iteration if the pilot user handles three consecutive real changes outside VIP by choice, does not return unprompted in the following period, or two reconciliations fail because employer rules are unknowable rather than fixable.

## Gate to Track B

Track B requires all of:

1. A completed Track A evidence pack with no unresolved safety pause.
2. A named second user from another employer and provider, a rate card, at least one payslip and a dated pilot commitment.
3. A written go/no-go decision accepting the security, verification and background-processing cost of mailbox access and notifications.

## Track B: product build

- Provider-neutral email/calendar connector contracts and configurable employer terminology.
- Read-only OAuth with incremental retrieval, explicit consent, encrypted server-side tokens, disconnection, deletion, export and recovery.
- Deterministic interpretation checks; uncertain messages always require review. AI output never writes financial records directly.
- Versioned integer-pence calculation rules recorded against each forecast.
- Full lifecycle: offered, planned, confirmed, worked, cancelled and disputed.
- Confirm-every-change at launch. Any later automation is opt-in per trusted sender, measured and reversible.
- CI blocks secret/personal-data leaks, dependency risk and reduced calculation fixtures. Human review is mandatory for money, OAuth, storage and personal-data changes.
- A second user must onboard unassisted and reconcile a payslip before product acceptance.

## Explicitly deferred

Net pay, broader childcare scenarios, dashboards beyond decision evidence, generic profiles beyond the second-user test, speculative connectors, background mailbox processing, automatic application and unaudited cloud sync.

## Immediate unresolved evidence

- Employer rounding point and break allocation across bands.
- Contract source for holiday accrual.
- Cutoff and payday confirmation from two payslips.
- Whether the pilot calendar feed expands recurrence and exposes removals reliably.
- Whether a subscribed output calendar's refresh delay is acceptable.
- Whether zero-copy email ingestion is valuable enough to pay the Track B verification/security cost.
